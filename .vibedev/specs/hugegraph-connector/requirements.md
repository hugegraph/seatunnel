# HugeGraph 连接器需求文档

## 简介

Apache SeaTunnel 的 HugeGraph 连接器是一个数据集成组件，用于实现从关系型数据库到 Apache HugeGraph 图数据库的实时数据同步。该连接器主要服务于泛安全领域的图分析场景，包括攻击溯源、告警关联网络和进程链图等。连接器将复用 SeaTunnel 现有的 CDC 能力，专注于将 SeaTunnelRow 数据转换为 HugeGraph 的顶点和边结构。

## 需求列表

### 1. 核心连接器实现

**用户故事**：作为数据工程师，我希望能够实时同步 MySQL 数据到 HugeGraph，以便对关系型数据进行基于图的安全分析。

**验收标准**：
1.1. 系统应实现遵循 SeaTunnel v2 连接器规范的 HugeGraph Sink 连接器
1.2. 连接器应支持 HugeGraph 1.5 版本及兼容版本
1.3. 连接器应处理 SeaTunnelRow 数据流并写入 HugeGraph
1.4. 连接器应预留 Source 接口供后续实现
1.5. 系统应通过 graph_space 和 graph_name 参数支持多图空间

### 2. CDC 事件处理

**用户故事**：作为数据分析师，我希望增量数据变更能够自动同步，以保持图数据库与源系统的实时一致性。

**验收标准**：
2.1. 连接器应处理来自 SeaTunnel MySQL CDC 源的 CDC 事件
2.2. 系统应支持 INSERT 操作（优先级1-最高）
2.3. 系统应支持 UPDATE 操作（优先级2）
2.4. 系统应支持 DELETE 操作（优先级3）
2.5. 当删除顶点时，系统应级联删除所有关联的边
2.6. 连接器应处理 CDC 事件元数据，包括操作类型和时间戳

### 3. 数据映射配置

**用户故事**：作为开发者，我希望有灵活的配置选项来映射关系型数据到图结构，以适应不同的数据模型。

**验收标准**：
3.1. 系统应支持多个 MySQL 表映射到单个 VertexLabel
3.2. 系统应支持单个 MySQL 表映射到多个 VertexLabel
3.3. 连接器应允许配置从表主键到顶点 ID 的映射
3.4. 连接器应支持联合主键生成顶点 ID
3.5. 对于边映射，系统应支持从单表字段获取源和目标顶点 ID
3.6. 对于边映射，系统应支持从关联表获取源和目标顶点 ID
3.7. 边的方向应通过字段顺序配置（如"a,b"表示 a→b）
3.8. 系统应提供常见映射场景的配置模板

### 4. Schema 管理

**用户故事**：作为数据库管理员，我希望连接器能验证 schema 兼容性，以确保同步过程中的数据完整性。

**验收标准**：
4.1. 连接器应在写入前验证目标 VertexLabel 在 HugeGraph 中存在
4.2. 连接器应在写入前验证目标 EdgeLabel 在 HugeGraph 中存在
4.3. 系统应验证源和目标 schema 之间的数据类型兼容性
4.4. 如果 schema 验证失败，连接器应报告清晰的错误信息
4.5. 连接器在当前阶段不应自动创建 schema

### 5. 批量写入策略

**用户故事**：作为系统运维人员，我希望有高效的数据写入和可配置的批处理，以根据工作负载优化性能。

**验收标准**：
5.1. 连接器应支持批量写入，默认批次大小为 500 条记录
5.2. 连接器应支持基于时间窗口的批处理，默认窗口为 5 秒
5.3. 系统应在批次大小或时间窗口任一阈值达到时触发批量写入
5.4. 批次大小应可通过连接器参数配置
5.5. 时间窗口应可通过连接器参数配置
5.6. 连接器应优雅处理部分批次失败

### 6. 错误处理和可靠性

**用户故事**：作为运维工程师，我希望有健壮的错误处理和重试机制，以避免临时故障导致数据丢失。

**验收标准**：
6.1. 连接器应对瞬时故障实现自动重试
6.2. 系统应支持可配置的重试次数（默认：3）
6.3. 系统应支持可配置的重试延迟和指数退避
6.4. 所有重试失败的记录应记录详细错误信息
6.5. 连接器应保证成功处理记录的最终一致性
6.6. 系统应提供监控写入成功/失败率的指标

### 7. 配置接口

**用户故事**：作为用户，我希望有清晰直观的配置选项，以便轻松设置数据同步管道。

**验收标准**：
7.1. 连接器应接受符合 SeaTunnel 标准的 HOCON/JSON 格式配置
7.2. 配置应包含 HugeGraph 连接参数（主机、端口、图名、凭证）
7.3. 配置应包含顶点和边的映射规则
7.4. 系统应在启动时验证配置完整性
7.5. 连接器应为常见用例提供示例配置文件
7.6. 配置应支持敏感数据的环境变量替换

### 8. 文档和示例

**用户故事**：作为新用户，我希望有全面的文档和示例，以便快速理解和使用连接器。

**验收标准**：
8.1. 系统应提供涵盖所有配置选项的用户文档
8.2. 文档应包含至少 3 个完整的示例配置
8.3. 示例应涵盖：简单顶点同步、边同步和基于 CDC 的增量同步
8.4. 文档应解释关系型和图数据模型之间的映射
8.5. 连接器应包含内联代码注释以提高可维护性

### 9. 性能要求

**用户故事**：作为系统管理员，我希望连接器能高效处理数据量，避免同步成为瓶颈。

**验收标准**：
9.1. 连接器应能处理至少 100 万条记录而不出现内存问题
9.2. 系统应在安全的情况下支持并发写入
9.3. 资源使用应可配置（如连接池大小）
9.4. 连接器应提供性能指标（吞吐量、延迟）

### 10. 测试要求

**用户故事**：作为质量工程师，我希望有全面的测试覆盖，以确保连接器的可靠性和可维护性。

**验收标准**：
10.1. 连接器应包含代码覆盖率 >80% 的单元测试
10.2. 系统应包含使用 testcontainers 的集成测试
10.3. 测试应覆盖所有 CDC 事件类型（INSERT、UPDATE、DELETE）
10.4. 测试应验证级联删除功能
10.5. 连接器应包含完整数据管道的端到端测试

## 成功标准

- 成功使用 SeaTunnel 作业配置从 MySQL 同步数据到 HugeGraph
- 正确处理 CDC 事件，包括顶点的级联删除
- 提供覆盖常见用例的灵活映射配置
- 在增量同步期间保持数据一致性
- 稳定处理 100 万+ 记录
- 通过所有单元测试、集成测试和端到端测试

## 约束和假设

- HugeGraph schema 必须预先定义（第一阶段不自动创建）
- 依赖 SeaTunnel 现有的 MySQL CDC 源进行变更捕获
- 专注于数据同步，而非转换（转换由 SeaTunnel transform 层处理）
- 初始实现优先考虑功能完整性而非性能优化
- Source 实现推迟到第二阶段
